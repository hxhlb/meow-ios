Build 2026052103 (1.3.1)

新特性
• 隧道内存:把进程内 TCP/IP 栈从 netstack-smoltcp 换成
  lwip。在持续 TCP churn 压力下,PacketTunnel 扩展的常驻
  内存稳定在约 50 MiB,不再像之前那样持续爬到 140-200 MiB。
  iOS NetworkExtension 的 jetsam 上限约为 50 MB,旧曲线在
  重负载下会触发可避免的扩展重启;新曲线留出了余量。背景
  与压测数据见
  docs/INVESTIGATION-2026-05-16-stress-rss-netstack-smoltcp.md。

• 按需自动连接:默认关闭。设置 → 通用 新增"按需自动连接"
  开关。开启后,iOS 会在扩展被系统回收(内存/CPU 压力等)
  之后自动恢复隧道;关闭后,VPN 在你手动开启之前保持断开。
  之前的版本对所有人强制开启,出乎部分用户预料。

• 启动崩溃修复:日志初始化顺序问题制造了
  tracing → log → tracing 的递归回路,在部分场景下会把
  宿主 App 启动期的栈打穿(事件 BA547B1D-5D66)。已把
  try_init 换成显式 set_global_default 并加上线程局部的
  重入护栏,从结构上杜绝该回路。

• 内部调整:TCP accept cap 从 32 提高到 256,与 lwip 监听
  队列深度匹配 —— 突发压力下,旧 cap 是瓶颈;现在瓶颈在
  lwip 内部队列上。

测试关注点
1. 真机启动 / 重启
   • 应用进入主界面不崩溃。(上面修复的 bug 在前一版本对
     部分测试人员表现为 SIGSEGV 即时崩溃。)

2. 设置 → 通用 → "按需自动连接"
   • 全新安装 / 首次启动:开关应该是关闭的。
   • 打开开关,杀掉应用并重新打开。在 iOS 设置 → 通用 →
     VPN 与设备管理 → meow 里,"按需连接"应该是 ON;
     手动断 Wi-Fi 重连之后,隧道应该自动恢复。
   • 关闭开关,通过主界面手动断开。隧道在网络变化后
     不应自动恢复。

3. 持续流量内存裕度
   • 跑一段高负载会话:多标签浏览、视频流、连续测速。
     观察 设置 → 关于 → 内存(连接时每 5 秒刷新一次)。
   • 期望:数值停留在 30-60 MiB 区间,不应单调上升。
     如果看到越过 ~45 MiB 还在继续涨,请记录峰值数值
     和距离连接起点的时间。

4. 常规回归
   • 连接 / 断开 / 切换代理组 / 刷新订阅。不应出现新的
     崩溃或卡死。

已知问题
• 非 DNS UDP 转发仍未支持(QUIC/HTTP3 会降级到 TCP,通常
  无感)。
• 蜂窝 vs Wi-Fi 吞吐与 2026051901 一致 —— MTU 钳制行为
  未变。
